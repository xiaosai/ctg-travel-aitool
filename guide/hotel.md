# 酒店预订操作指南

**触发条件**：见 [SKILL.md](../SKILL.md) 中「一、需求识别与分流」的酒店触发示例。本指南在识别到用户要**预订或查询酒店**时按需加载。

---

## 必填字段收集原则（通用）

**不写死必填项**。在需要向后台发起请求时：
1. 查阅 [api/hotel.json](../api/hotel.json) 中对应 method 的 `parameters.required` 及 `properties`
2. 若用户未给出某必填字段，用自然语言提示用户填写（勿说「请提供某某参数」）
3. 每次只问一项缺项，待用户回复后再继续
4. 所有必填字段收集完整后再发起请求

**面向用户的话术**：对用户说的内容禁止出现「调用某某接口」「请求 API」「getPassengerList」等任何技术用语，一律使用「正在为您查询…」「请选择入住人」等业务话术。

---

## 核心步骤与数据流（Agent 内部参考，勿对用户说）

以下为内部执行顺序与数据依赖，**向用户展示时只用业务话术**（如「正在查询酒店」「正在预订」），不得出现接口名、API、method 等。

```
1. 获取城市列表 → 得到目标城市的 cityId
2. 查询酒店列表 → 使用 cityId、arrivalDate、departureDate
3. 查询酒店详情（房型列表）→ 使用 hotelId、arrivalDate、departureDate
4. 酒店验价 → 使用 resourceItemId、checkInDate、checkOutDate、roomCount
5. 查询入住人列表 → 若为空则先新增入住人再继续
6. 创建酒店订单 → 使用 resourceItemId、travelerList、price 等
7. 轮询订单状态 → 预订完成后引导用户支付
```

## 完整交互流程

### 第一阶段：收集基本信息（城市、入住日期、离店日期）

**内部**：先获取城市列表，用于后续酒店查询。根据用户输入的城市，从城市列表中匹配 `cityId`。

**请求参数（酒店场景）**：以 [api/hotel.json](../api/hotel.json) 为准，酒店入参示例：`domesticType=1`、`resourceType=1`（酒店=1）。具体见 api 文档。

### 🔧 Python 调用命令

**命令格式（cmd）**：
```bash
python scripts/apiexe.py call --method cityList --arg "{\"domesticType\": 1, \"resourceType\": 1}"
```

**参数说明**：
- `domesticType`: 国内/国际类型，默认国内 1（国内）2（国际）
- `resourceType`: 资源类型，酒店传 1（飞机=0, 酒店=1, 火车=2, 门票=3）

**对用户的说法**（自然询问，不要机械填表）：

```
好的，我来帮您预订酒店！
📍 在哪个城市预订？
📅 什么时候入住？（可以说"明天"、"本周六"、"3月15日"等）
📅 什么时候离店？（可以说"住两晚"、"后天"、"3月17日"等）
```

**智能识别日期**：

| 用户输入 | 转换结果 |
|---------|---------|
| 今天/今日 | 当前日期 |
| 明天/明日 | 当前日期+1 |
| 后天、大后天 | 对应计算 |
| 下周一~下周日、这周一~这周日 | 计算对应日期 |
| 3月15日/3月15号/3-15 | 解析为 YYYY-MM-DD（如 2026-03-15） |
| 住两晚/住三天 | 根据入住日期计算离店日期 |
| 5天后 | 当前日期+N |

**智能计算离店日期**：
- 用户说「住两晚」→ 离店日期 = 入住日期 + 2天
- 用户说「住三天」→ 离店日期 = 入住日期 + 3天

**城市匹配**：用户说出城市名后，从城市列表返回结果中查找对应城市的 `cityId`，用于下一步查询酒店。

### 第二阶段：展示酒店选项

查询酒店列表，必填字段按 [api/hotel.json](../api/hotel.json)，缺则向用户自然追问（勿说「缺少 cityId 参数」等）。

**参数映射**（来自城市列表结果）：
- `cityId`：城市列表返回的 cityId
- `arrivalDate`：入住日期，**格式 yyyy-MM-dd**（如 2026-03-15）
- `departureDate`：离店日期，**格式 yyyy-MM-dd**
- `key`：搜索关键词（酒店名、品牌、地标等）
- `page`：页码，默认 1
- `pageSize`：每页数量，默认 10
- `entranceSource`：入口来源，0-新旅行 1-疗休养

### 🔧 Python 调用命令

**命令格式（cmd）**：
```bash
python scripts/apiexe.py call --method hotel.search --arg "{\"cityId\": 4, \"arrivalDate\": \"2026-03-15\", \"departureDate\": \"2026-03-17\", \"page\": 1, \"pageSize\": 10}"
```

**关键参数说明**：
- `cityId`: 城市ID（从城市列表获取）
- `arrivalDate`: 入住日期，格式 yyyy-MM-dd（如 2026-03-15）
- `departureDate`: 离店日期，格式 yyyy-MM-dd（如 2026-03-17）
- `key`: 搜索关键词（可选，如酒店名、品牌名）

**注意事项**：
- 日期格式必须为 yyyy-MM-dd，带横杠
- `entranceSource` 默认为 0（新旅行）

**展示格式示例**（接口返回后润色展示，一行一行展示，勿用表格）：

```

📅 2026年3月15日 - 3月17日（2晚） 北京
```

**智能筛选和推荐**：

### 第五阶段：酒店验价（可选）

在创建订单前，可调用验价接口确认价格和房态。必填字段按接口文档。

**入参**：
- `resourceItemId`：房型ID（从酒店详情获取）
- `checkInDate`：入住日期，格式 yyyy-MM-dd
- `checkOutDate`：离店日期，格式 yyyy-MM-dd
- `roomCount`：房间数，默认 1
- `latestCheckInTime`：最晚到店时间
- `totalAmount`：总金额

### 🔧 Python 调用命令

**命令格式（cmd）**：
```bash
python scripts/apiexe.py call --method hotel.validatePrice --arg "{\"resourceItemId\": \"RES_123456789\", \"checkInDate\": \"2026-03-15\", \"checkOutDate\": \"2026-03-17\", \"roomCount\": 1}"
```

**关键参数说明**：
- `resourceItemId`: 资源项ID（房型ID，从酒店详情获取）
- `checkInDate`: 入住日期，格式 yyyy-MM-dd
- `checkOutDate`: 离店日期，格式 yyyy-MM-dd
- `roomCount`: 房间数，默认 1

**验价失败处理**：
- 若返回「房间已满」等错误，提示用户选择其他房型或日期
- 若价格变动，展示最新价格让用户确认

---

### 第六阶段：下单预订

创建酒店订单。入参以 [api/hotel.json](../api/hotel.json) 为准，**按文档结构传参**：

**顶层必传**：memberId、phoneNumber、orderSource、checkInDate、checkOutDate、roomNum、resourceItemId、travelerList、price

**travelerList[]** 每条需包含：
- passengerId：入住人ID（从入住人列表获取）
- name：入住人姓名
- identityNo：证件号码
- phoneNumber：手机号码

**重要**：订单创建成功后处于**预订中**状态，**必须**进入第七阶段轮询订单状态，根据结果再决定提示用户支付或失败重试，切勿未轮询就提示用户支付。

### 🔧 Python 调用命令

**命令格式（cmd）**：
```bash
python scripts/apiexe.py call --method hotel.createOrder --arg "{\"memberId\": \"15d6676f6be54d5099b106abeeecfcd6\", \"phoneNumber\": \"15000000000\", \"orderSource\": 0, \"checkInDate\": \"2026-03-15\", \"checkOutDate\": \"2026-03-17\", \"roomNum\": 1, \"resourceItemId\": \"RES_123456789\", \"price\": 398, \"travelerList\": [{\"passengerId\": 399, \"name\": \"张三\", \"identityNo\": \"420102199007015297\", \"phoneNumber\": \"15000000000\"}]}"
```

**关键参数说明**：
- `checkInDate`、`checkOutDate`: 入住/离店日期，格式 yyyy-MM-dd
- `roomNum`: 房间数
- `resourceItemId`: 资源项ID（房型ID，从酒店详情获取）
- `price`: 单价（单间每晚价格）
- `travelerList`: 入住人列表，从入住人列表中选取

**注意事项**：
- 订单创建成功后会返回 `orderBaseId`，用于后续轮询订单状态

---

### 第七阶段：订单状态轮询（必执行）

下单成功后订单处于预订中，**必须**在有限次数内轮询订单状态，直到得到明确结果。轮询时使用的订单号从创建订单返回的 `orderBaseId` 获取。

- **轮询间隔**：10 秒
- **最大轮询次数**：6 次（最多等待约 1 分钟）

**状态码与处理**：

| 状态码 | 含义 | Agent 处理 |
|--------|------|-------------|
| `10` | 处理中 | 继续轮询，可提示「正在为您预订，请稍候…」 |
| `12` | 预订完成 | 进入第八阶段，展示成功反馈并提醒支付 |
| `11` | 预订失败 | 进入第八阶段，展示失败反馈并引导重新预订 |
| 超时 | 轮询 6 次仍为 10 | 告知「预订超时，请稍后查询订单状态」 |

### 🔧 Python 调用命令

**命令格式（cmd）**：
```bash
python scripts/apiexe.py call --method getOrderStatus --arg "{\"orderBaseId\": \"HRO202603101234567890\"}"
```

**参数说明**：
- `orderBaseId`: 订单号（从创建订单返回获取）

### 第九阶段：取消订单（可选）

当用户表达取消酒店订单意愿时，执行取消订单。必填为订单号（orderBaseId），按 [api/hotel.json](../api/hotel.json) 文档，缺则向用户确认订单号。

**场景**：用户想预订其他酒店/房型，需先取消当前未支付订单。

**流程**：
1. 向用户确认：「检测到您有未支付的订单，是否需要取消后重新预订？」
2. 用户同意后执行取消
3. 取消成功后继续新流程

### 🔧 Python 调用命令

**命令格式（cmd）**：
```bash
python scripts/apiexe.py call --method hotel.cancelOrder --arg "{\"orderBaseId\": \"HRO202603101234567890\", \"orderType\": 1, \"cancelReason\": \"不需要了\", \"terminalCode\": \"APP\"}"
```

**参数说明**：
- `orderBaseId`: 订单号（必填）
- `orderType`: 订单类型，酒店传 1
- `cancelReason`: 取消原因
- `terminalCode`: 终端类型，如 "APP"

**确认话术示例**（润色展示，一行一行展示，勿用表格）：

```
⚠️ 检测到您有未支付订单

如需预订其他酒店，需先取消当前订单。

是否取消该订单？(y/n)
```

**取消成功**：「✅ 已为您提交取消申请，订单取消后将自动释放房间。」

## 用户体验

### 对话风格

✅ **友好自然**：「好的，我来帮您查询北京的酒店！」
❌ **机械生硬**：「请提供城市、入住日期和离店日期。」

### 后台返回结果润色（重要）

收到后台返回后，**不要直接原样输出**，应润色成用户可读的文案：

- **使用符号**：🏨 🛏️ 👥 📋 💰 ⏰ ✅ ❌ ⚠️ 等增强可读性
- **一行一行展示**：每条信息单独一行，**勿用表格、框线**（`│` `├` `┌` `└` 等），避免在不同界面展示错乱
- **适当留白**：信息块之间空行分隔，避免拥挤
- **引导语**：在信息块末尾加简短操作提示

### 信息展示

✅ **结构化清晰**：分块展示酒店、房型、入住人、费用
❌ **信息堆砌**：整段文字挤在一起

### 错误处理

✅ **提供解决方案**：列出可选方案，让用户选择
❌ **只报错**：仅提示「错误：系统异常」
